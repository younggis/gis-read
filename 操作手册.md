## 支持的格式

| 格式 | 扩展名 | 解析 | 写入 | 说明 |
| --- | --- | --- | --- | --- |
| **Shapefile** | `.shp` + `.dbf`/`.shx`/`.prj`/`.cpg` | ✅ | ✅ | ESRI 主流矢量格式；写出要求单一几何族 |
| **MapInfo TAB** | `.tab` + `.dat`/`.map`/`.id`/`.cpg` | ✅ | ✅* | 写出依赖 GDAL `ogr2ogr`；支持 TAB 字符集、中文属性值和常见 legacy 线对象读取 |
| **GeoJSON** | `.geojson`, `.json` | ✅ | ✅ | OGC 开放标准 |
| **KML** | `.kml` | ✅ | ✅ | Keyhole Markup Language (Google Earth) |
| **GPX** | `.gpx` | ✅ | ✅ | GPS Exchange Format (waypoint/track/route) |
| **TopoJSON** | `.topojson` | ✅ | ✅ (→ GeoJSON) | D3 共享弧拓扑 |
| **CZML** | `.czml` | ✅ | ✅ (→ GeoJSON) | Cesium 时间动态场景 |
| **CSV (WKT)** | `.csv` | ✅ | ✅ | 含 WKT 列或 lat/lng 列；写出 `wkt` 几何列 |
| **ESRI JSON** | `.json` (带 `spatialReference`) | ✅ | ✅ | ArcGIS REST/JSON 几何 |
| **MapInfo MIF** | `.mif` + `.mid` | ✅ | ✅ | 旧版 MapInfo 文本交换格式 |

格式自动通过文件扩展名 + 魔数嗅探识别；`.json` 文件会进一步检查 `type` / `spatialReference` / `czml` 等字段做二次分类。

## 支持的坐标系

| ID 别名 | 名称 | 加密？ | 用途 |
| --- | --- | --- | --- |
| `WGS84` / `4326` / `EPSG:4326` | WGS 84 地理坐标 | 否 | 国际通用、GPS |
| `WebMercator` / `3857` / `EPSG:3857` | Web 墨卡托投影 | 否 | 在线地图（Google/百度/高德外网） |
| `CGCS2000` / `4490` / `国家2000` | 中国国家 2000 坐标系 | 否 | 中国官方大地坐标（与 WGS84 民用一致） |
| `GCJ02` / `火星` / `火星坐标系` | 火星坐标系 (国测局) | 是 | 高德/腾讯/谷歌中国地图 |
| `BD09` / `百度` / `百度坐标系` | 百度坐标系 | 是 | 百度地图专用 |
| 任意 `EPSG:xxxx` | proj4 支持的坐标系 | 否 | 通过 `proj4` 库动态注册 |

### 加密坐标系说明

中国大陆的 `GCJ02` (火星) 和 `BD09` (百度) 是非线性加密坐标：
- `GCJ02` 在 WGS84 基础上加了几十米到几百米的伪随机偏移。
- `BD09` 在 GCJ02 基础上再叠加一层加密。
- 两者的反向变换（加密→WGS84）需要迭代求解，本工具通过 5 次牛顿迭代实现，约 1 cm 精度。
- **仅在中国大陆境内 (经度 73.66-135.05, 纬度 3.86-53.55)** 应用加密；区域外自动直通。

## 安装与构建

```bash
# 依赖
node >= 18

# 安装
npm install

# 构建
npm run build

# 直接通过 tsx 运行（无需 build）
npm run dev -- info data/lakes.shp
```

构建产物在 `dist/`，入口为 `dist/cli.js`，可通过 `node dist/cli.js` 调用，或在 `package.json` 中已经声明 `bin: gis`，全局安装后可直接用 `gis` 命令。

## 子命令详解

### `info <file>`

显示文件元信息：格式、要素数、坐标系、包围盒、shape 类型等。

```bash
node dist/cli.js info data/lakes.shp
# File:      F:\gistool\gis-read\data\lakes.shp
# Size:      9.1 MB
# Format:    shapefile
# Name:      lakes
# Features:  1225
# CRS:       GEOGCS
# BBox:      97.60736916, 26.18485398, 108.00993096, 33.78606597
# shapeTypeName Polygon
# recordCount 1225
```

### `detect <file>`

仅输出检测到的格式（`shapefile` / `geojson` / `kml` / `tab` / `gpx` / `topojson` / `czml` / `csv` / `esrijson` / `mif` / `unknown`）。

```bash
node dist/cli.js detect data/lakes.kml
# kml
```

### `parse <file>`

把文件解析为 JSON 打印到 stdout，常用于管道。

```bash
# 完整输出
node dist/cli.js parse data/lakes.geojson > parsed.json

# 只看前 5 个要素
node dist/cli.js parse data/lakes.geojson --limit 5

# 单行紧凑输出（适合管道到 jq）
node dist/cli.js parse data/lakes.geojson --no-pretty | jq '.features[0].properties'
```

### `convert <input> -o <output>`

跨格式转换，可选重投影。

```bash
# 基本转换
node dist/cli.js convert data/lakes.shp -o output/lakes.geojson
node dist/cli.js convert data/lakes.tab -o output/lakes-tab.geojson
node dist/cli.js convert data/lakes.geojson -o output/lakes.kml
node dist/cli.js convert data/lakes.geojson -o output/lakes.esrijson -t esrijson
node dist/cli.js convert data/lakes.geojson -o output/lakes.csv
node dist/cli.js convert data/points.geojson -o output/points.shp -t shapefile
node dist/cli.js convert data/lakes.geojson -o output/lakes.mif
node dist/cli.js convert data/lakes.geojson -o output/lakes.tab -t tab  # 需要 GDAL ogr2ogr

# 指定格式（可省略）
node dist/cli.js convert data/lakes.tab -f tab -t geojson -o output/lakes.geojson

# 带重投影
node dist/cli.js convert data/lakes.shp -o output/lakes.geojson \
  --from-crs WGS84 --to-crs WebMercator

# 控制精度
node dist/cli.js convert data/lakes.geojson -o output/lakes.kml --precision 4
```

当前可写出的目标格式是 `geojson` / `kml` / `gpx` / `esrijson` / `csv` / `mif` / `shapefile` / `tab`。其中 GPX 只表达 waypoint 和 track/route，适合 Point / LineString / MultiLineString；Polygon / MultiPolygon 会被跳过。Shapefile 写出要求同一个 bundle 只包含一种几何族；MapInfo TAB 写出委托给 GDAL `ogr2ogr`，未安装 GDAL 时请改写 MIF。

### `stream <input> -o <output>`

面向大 GeoJSON 的恒定内存转换。输入必须是 GeoJSON，输出根据扩展名识别为 `.geojson` / `.kml` / `.gpx`。

```bash
# 大 GeoJSON 转 KML
node dist/cli.js stream data/lakes.geojson -o output/lakes.kml

# 大 GeoJSON 转 GeoJSON，可同时重投影
node dist/cli.js stream data/lakes.geojson -o output/lakes-3857.geojson \
  --from-crs WGS84 --to-crs WebMercator

# convert 也支持同一条流式路径
node dist/cli.js convert data/lakes.geojson -o output/lakes.kml --stream
```

回归验证：`node dist/cli.js stream data/lakes.geojson -o output/user-stream-lakes.kml` 当前输出约 13 MB，包含 1225 个 `<Placemark>`，不是空文件。若输出为空，先确认已经执行 `npm run build`，再检查 stderr 的错误日志。

### `crs <file> --to <crs>`

原地或输出到新文件地重投影 GeoJSON。

```bash
# 原地（覆盖输入）
node dist/cli.js crs data/lakes.geojson --to WebMercator

# 输出到新文件，指定源坐标系
node dist/cli.js crs input.geojson --to GCJ02 --from WGS84 -o output-gcj.geojson
```

### `crs-info <crs>`

```bash
node dist/cli.js crs-info BD09
# ID:         BD09
# Name:       BD-09 (百度坐标系)
# Encrypted:  true
```

## 字符编码自动识别

中国 GIS 数据最常见的问题是**中文乱码**。本工具实现了**三层编码自动识别机制**：

### 识别优先级

| 优先级 | 来源 | 说明 |
| --- | --- | --- |
| 1 | `.cpg` 文件 | Shapefile / MapInfo TAB 标准 sidecar，明确指定字符集（如 `UTF-8`、`GBK`、`Big5`） |
| 2 | `.tab` 文本中的 `!charset` 行 | MapInfo TAB 格式头自带 charset 提示（如 `WindowsLatin1`、`Neutral`） |
| 3 | 内容启发式探测 | 对样本数据进行 8 种编码试解码并打分：CJK 字符命中、mojibake 惩罚、常见 GIS 中文词加权 |
| 4 | dBASE 语言驱动字节 | `.dbf` 第 29 字节的 Visual FoxPro 语言驱动码（最不可靠，仅作最终回退） |

### 支持的编码

`utf-8` / `utf-8` / `gbk` / `gb18030` / `gb2312` / `cp936` / `big5` / `cp950` / `shift_jis` / `euc-kr` / `cp1250`-`cp1258` / `cp437`-`cp857` / `windows-1252` / `latin1` 等。

### 验证

```bash
# 查看实际使用的编码（含来源）
node dist/cli.js info data/lakes.shp
# ...
# encoding   utf-8 (cpg)        ← 来自 .cpg 文件

node dist/cli.js info data/lakes.tab
# ...
# encoding   utf-8 (cpg)        ← 来自 .cpg 文件
```

### 三层兜底的效果

样例数据 `lakes.shp` / `lakes.tab` 即使在没有 `.cpg` 文件时也**自动识别为 UTF-8** 并正确解码中文（"茶园沟水库"）。如果数据是 GBK / Big5 / GB18030 编码，工具也会通过启发式打分正确识别。

### 编程式使用

```ts
import { detectEncoding, decodeStringField, decoderFor, readCPG } from 'gis-read';

// 1. 探测
const buf = Buffer.from('北京市海淀区', 'gbk');
const enc = detectEncoding(buf);  // 'gb18030' 或 'gbk'
const dec = decoderFor(enc);
console.log(dec(buf));  // 北京市海淀区

// 2. 解码 .dbf 字段
const fieldBytes = Buffer.from('北京水库   \x00\x00', 'binary');
const clean = decodeStringField(fieldBytes, decoderFor('utf-8'));
// '北京水库'

// 3. 读取 .cpg
const hint = readCPG('/path/to/file.cpg');  // 'utf-8' or null
```

### 常见乱码场景对照

| 现象 | 原因 | 解决方案 |
| --- | --- | --- |
| `ä¸­æ–‡` 形式乱码 | UTF-8 字节被当作 Latin1 解码 | 工具已自动识别为 UTF-8 重新解码 |
| `?` 替换字符 | GBK 编码用 UTF-8 解码 | 工具探测到 CJK 字符后切到 GBK/GB18030 |
| 繁体中文乱为 `` | Big5 被误判为 GBK | 工具会区分 Big5 (0xA4 0xA4 起始) 与 GBK (0xB0 0xA1 起始) |
| `???` 问号 | 字符集不在探测列表 | 在 `.cpg` 中显式指定，或自行用 `decoderFor('xxx')` 手动解码 |
| 部分字段对部分错 | 数据本身混编（如部分 UTF-8 + 部分 GBK） | 建议先用 ogr2ogr 统一编码后再处理 |

## 编程式使用

```ts
import {
  parseFile,
  parseShapefile,
  parseTAB,
  parseGeoJSON,
  parseKML,
  parseGPX,
  parseTopoJSON,
  parseCZML,
  parseCSV,
  parseEsriJSON,
  parseMIF,
  writeGeoJSON,
  writeKML,
  writeGPX,
  writeEsriJSON,
  writeCSV,
  writeMIF,
  writeShapefile,
  writeTAB,
  writeFile,
  detectFormat,
  type Format,
} from 'gis-read';
import {
  transformFeatures,
  transformCoord,
  wgs84ToGCJ02,
  gcj02ToBD09,
  getCRS,
} from 'gis-read';

// 1. 直接解析
const shp = parseShapefile('data/lakes.shp');
console.log(shp.features[0].properties);
// { OBJECTID: 1, Name: '茶园沟水库', SHAPE_Area: 2.157e-5, ... }

// 2. 工厂解析（自动检测）
const result = parseFile('data/lakes.kml');

// 3. CRS 转换
const [lng, lat] = wgs84ToGCJ02(116.391, 39.907); // 天安门
console.log(lng, lat); // 116.397, 39.908 (火星坐标)

const bd = gcj02ToBD09(lng, lat); // 火星 → 百度
console.log(bd); // 百度坐标

// 4. 批量重投影
const geo = parseGeoJSON(fs.readFileSync('beijing.geojson'));
transformFeatures(geo.features, 'WGS84', 'GCJ02');
// geo.features 的 geometry 已被原地更新

// 5. 写入
fs.writeFileSync('out.geojson', writeGeoJSON(geo, { precision: 4 }));
fs.writeFileSync('out.kml',    writeKML(geo, { precision: 4 }));
fs.writeFileSync('out.csv',    writeCSV(geo, { precision: 4 }));
writeMIF(geo, { outputPath: 'out.mif', precision: 4 });          // 生成 out.mif + out.mid
writeShapefile(parseFile('points.geojson'), { outputPath: 'points.shp' }); // 生成 .shp/.shx/.dbf/.cpg
writeTAB(geo, { outputPath: 'out.tab' });                        // 需要 GDAL ogr2ogr
writeFile(geo, 'out.csv', 'csv', { precision: 4 });               // 统一写入入口
```

返回的 `ParseResult`：

```ts
interface ParseResult {
  name?: string;          // 数据集名
  features: Feature[];    // GeoJSON Feature 数组
  crs?: CRS;              // 坐标系描述
  bbox?: BBox;            // 包围盒 [minX, minY, maxX, maxY]
  meta?: Record<string, unknown>;  // 格式特定元数据
}
```

## 格式说明

### Shapefile (`.shp`)

- 必需文件：`.shp`（几何）+ `.dbf`（属性）
- 可选文件：`.shx`（索引）、`.prj`（投影 WKT）、`.cpg`（dbf 字符集）
- 支持的几何类型：`Point` / `PointM` / `PointZ`、`PolyLine` / `PolyLineM` / `PolyLineZ`、`Polygon` / `PolygonM` / `PolygonZ`、`MultiPoint` / `MultiPointM` / `MultiPointZ`
- Polygon 的 outer/hole 通过环的有向面积（signed area）自动判定
- `.prj` 投影以 WKT 形式保留在 `crs.properties.name`
- 写出时生成 `.shp/.shx/.dbf/.cpg`；同一个 Shapefile bundle 只能包含一种几何族（点、线或面），混合几何会报错
- dBASE 字段名会按 Shapefile 限制压缩到 10 个字符以内，复杂属性值会转为 JSON 字符串

### MapInfo TAB (`.tab`)

- 必需文件：`.tab`（文本头）+ `.dat`（dBASE III 属性）+ `.id`（索引）
- 几何：`.map`（v300 二进制）
- 字段类型按 `.tab` 文本头的 `Definition Table` 解析（Float / Integer / LargeInt / Char / Date / Logical）
- 几何提取：当前版本对标准对象类型 (Point=1, Line=2, Polyline=3, Region=4) 完整支持；常见 legacy 线对象会解析为 `LineString` / `MultiLineString`；其他 v300 私有类型 (0x0D 等) 可能返回 `null`（属性仍然完整）
- 如需完整 MapInfo 几何，建议先用 ogr2ogr 转 GeoJSON 再处理
- 写出时调用本机 GDAL `ogr2ogr` 生成 MapInfo TAB bundle；如果没有安装 GDAL，会提示改用 MIF

### KML (`.kml`)

- XML 格式，命名空间 `http://www.opengis.net/kml/2.2`
- 自动解析 `<Placemark>` 的 `<name>`、`<description>`、`<ExtendedData><Data>`、`<SchemaData><SimpleField>`
- 支持的几何：Point / LineString / Polygon / MultiGeometry（自动聚合为 MultiPoint / MultiLineString / MultiPolygon）

### GPX (`.gpx`)

- `<wpt>` → Point feature
- `<trk><trkseg>` → LineString feature（多段生成多个 features）
- `<rte>` → LineString feature
- 高程 `<ele>` 保留为坐标的第三分量

### TopoJSON (`.topojson`)

- 自动展开共享弧（arcs）到具体坐标
- 支持 `transform.scale/translate` 反量化
- 输出时转写为标准 GeoJSON FeatureCollection

### CZML (`.czml`)

- 处理 entity packets，document packet 跳过
- 支持 `position` / `polyline.positions` / `polygon.positions`
- 自动解析 `{ cartographicDegrees: [...] }` 引用形式

### CSV (`.csv`)

- 优先识别 `wkt` / `WKT` / `geometry` / `geom` / `the_geom` / `shape` 列
- 若无 WKT 列，回退到 `lat`+`lon` 列（自动识别多种列名）
- 支持引号、转义、UTF-8
- 写出时保留所有属性列，并新增 `wkt` 几何列；如果已有 `wkt` 属性，则几何列使用 `geometry`

### ESRI JSON (`.json`)

- 识别 `geometryType` / `spatialReference` / `features` 结构
- 写出时使用 `rings` (polygon) / `paths` (line) / `points` (multipoint)
- CRS 转换为 `EPSG:xxxx` 形式

### MapInfo MIF (`.mif`)

- 必需文件：`.mif`（文本几何）+ `.mid`（属性）
- 支持 Point / Line / Polyline / Region（自动聚合为 Polygon / MultiPolygon）
- 写出时生成 `.mif` 和同名 `.mid`；属性类型会从值推断为 Integer / Float / Logical / Char

## 坐标系详解

### 标准投影（proj4 实现）

- `WGS84` ↔ `WebMercator`：正反变换无损
- `WGS84` ↔ `CGCS2000`：完全相同（民用精度 < 1cm）
- 任意 `EPSG:xxxx`：通过 proj4 动态注册

### GCJ-02 (火星)

- 在中国境内对 WGS84 经纬度加几十至几百米偏移
- 反向变换通过牛顿迭代 5 次，精度约 1cm
- 区域外自动直通（无变换）

### BD-09 (百度)

- 在 GCJ-02 基础上加二次变换 + 0.0065/0.006 常数偏移
- 转换链路：`WGS84 ⇄ GCJ02 ⇄ BD09`
- 反向：`BD09 → GCJ02 → WGS84`

### 精度与限制

- 加密坐标不是严密可逆的，多次往返累计误差 < 1 米
- 仅支持 `lat/lng` 形式的加密；不直接支持投影坐标的 GCJ-02 加密（请先反投影到地理坐标再调用）

## 大文件 / 性能 / 日志

### 大文件处理

本工具实现了 **GeoJSON 的流式解析**，可以在远小于文件大小的内存下处理几个 GB 的大文件。

**流式 vs 内存式自动选择**：

```typescript
import { parseGeoJSONAuto, parseGeoJSONStream } from 'gis-read';

// 自动选择：≤ 256 MB 用内存模式（更快），> 256 MB 切到流式
const result = await parseGeoJSONAuto('huge.geojson');

// 强制流式
for await (const feature of parseGeoJSONStream('huge.geojson')) {
  // 处理 / 转换 / 写入到下游
}
```

**流式转换命令**：

```bash
# 将大 GeoJSON 转 KML（流式读取 + 写入，恒定内存）
node dist/cli.js stream big.geojson -o out.kml

# 转换时同时重投影
node dist/cli.js stream big.geojson -o out.kml --from-crs WGS84 --to-crs WebMercator

# 也可以用 convert --stream
node dist/cli.js convert big.geojson -o out.geojson --stream
```

支持的流式输入：**GeoJSON**。支持的流式输出：**GeoJSON / KML / GPX**。KML 流式写出复用内存模式的几何序列化逻辑，支持 Point / LineString / Polygon / MultiGeometry；GPX 流式写出只保留 Point 和 LineString。

**内存占用分析**：

| 模式 | 内存占用 | 适用规模 |
| --- | --- | --- |
| 内存模式（`parseGeoJSON`） | 文件大小 × 2-3 | ≤ 256 MB |
| 流式模式（`parseGeoJSONStream`） | 单个最大 feature 大小 | 任意大小（受磁盘限制） |

### 性能建议

| 优化项 | 说明 |
| --- | --- |
| **多 GB GeoJSON** | 务必用 `stream` / `--stream` 或 `parseGeoJSONStream`，否则 OOM |
| **重复调用** | 复用 `proj4` 投影函数以避免重复注册 |
| **GPX/KML 大文件** | 当前是同步读取，超大文件建议先用 `shpjs` 切分 |
| **写入 KML/GPX** | `writeKML` / `writeGPX` 是内存模式；用 CLI 的 `stream` 子命令做流式 |
| **写入 Shapefile** | 先按点/线/面拆分；单个 Shapefile bundle 不支持混合几何族 |
| **写入 MapInfo TAB** | 需要本机安装 GDAL `ogr2ogr`；无 GDAL 时优先写出 MIF |
| **多核 CPU** | 多次并行调用 CLI（Node.js 单进程跑单核） |

### 日志系统

所有 CLI 输出都经过统一的 `Logger` 模块，结构化格式：

```
2026-06-06T08:00:00.000Z INFO  [gis] Converted geojson -> kml features=1225 input="data/lakes.geojson" output="out.kml" elapsed_ms=643
```

字段：

| 字段 | 含义 |
| --- | --- |
| 时间戳 | ISO 8601 精确到毫秒 |
| 级别 | `DEBUG` / `INFO` / `WARN` / `ERROR` / `SILENT` |
| `[tag]` | 组件名（默认 `gis`） |
| 消息 | 人类可读描述 |
| `key=value` | 结构化字段，便于 grep / jq |

**全局选项**：

```bash
# 调整日志级别
node dist/cli.js --log-level debug info data/lakes.shp
node dist/cli.js --log-level silent convert big.geojson -o out.kml

# 写入到文件
node dist/cli.js --log-file run.log convert big.geojson -o out.kml
```

**编程式使用**：

```typescript
import { Logger, log } from 'gis-read';
log.setLevel('debug');
log.info('start', { file: 'big.geojson', size: 1_000_000 });
const done = log.startTimer('convert');
// ... do work ...
done('convert complete', { features: 1225 });
// 输出: ... INFO [gis] convert complete features=1225 elapsed_ms=523
```

### 异常处理

所有 CLI 子命令都通过 `withErrorBoundary` 包装：

- 任何抛出的异常被捕获，**结构化记录到 stderr**（包含 `name` / `message`）
- 退出码为 1（成功为 0）
- `GIS_DEBUG=1` 环境变量会打印完整堆栈
- 致命错误总是有清晰的中文/英文提示，不会让用户看到 Node.js 原始堆栈

**编程式包装**：

```typescript
import { withErrorBoundary } from 'gis-read';
const exitCode = await withErrorBoundary('myCommand', async () => {
  // throw new Error('boom');
});
// exitCode === 1; error logged
```

## 故障排查

### 1. `Error: Unknown / unsupported format for: <file>`

文件格式无法识别。检查：
- 文件扩展名是否正确
- `.shp` 是否能读取 4 字节魔数 9994 (0x0000270A)
- 用 `detect <file>` 查看自动检测结果

### 2. TAB 文件 `geometry: null`

常见 legacy 线对象已支持；如果仍出现 `geometry: null`，通常是 v300 `.map` 二进制的未支持私有 record 类型（如 0x0D）。属性仍然完整，解决方案：
- 在 QGIS 中导出为 GeoJSON
- 或使用 ogr2ogr：`ogr2ogr -f GeoJSON out.geojson in.tab`

### 3. SHP / TAB 字段值不对

- 检查 `.cpg` 文件是否指定了正确的字符编码（默认 Latin1，中文应使用 UTF-8 或 GBK）
- 在 TAB 中，`.tab` 文本头的字段类型优先于 `.dat` 的 dBASE 类型

### 4. CRS 转换精度异常

- 确认输入文件本身是 WGS84（其它坐标系传入 `--from-crs` 时会先转 WGS84 再到目标，可能累积误差）
- 加密坐标往返 5 次以上会累积到数米

### 5. `Buffer offset out of range`

通常是文件截断。检查：
- SHP 文件是否完整下载（.shp 头部有 24..27 字节声明的文件长度）
- TAB 的 .map / .dat / .id 是否一一对应

### 6. `stream` 输出为空或只有 XML 头

先确认运行的是最新构建产物：

```bash
npm run build
node dist/cli.js stream data/lakes.geojson -o output/check.kml --log-level debug
```

正常情况下 `data/lakes.geojson` 会输出 1225 个 `<Placemark>`。如果日志中 `features=0`，说明输入不是标准 GeoJSON `FeatureCollection`，或 `features` 数组无法被流式解析；可先用 `node dist/cli.js parse <file> --limit 1` 验证文件结构。若输出 GPX，请确认输入几何是 Point 或 LineString，面数据会被跳过。

### 7. 写出 TAB 时提示缺少 GDAL/OGR

MapInfo TAB 写出委托给 GDAL 的 `ogr2ogr`：

```bash
ogr2ogr --version
```

如果命令不存在，请安装 GDAL 并确认 `ogr2ogr` 在 `PATH` 中；或者改用纯 TypeScript 写出的 MapInfo MIF：

```bash
node dist/cli.js convert in.geojson -o out.mif
```
